# Claude Code Monitor v1 백로그

- 문서 상태: Draft
- 작성일: 2026-03-08
- 기준 문서: [prd-v1.md](./prd-v1.md), [ia-v1.md](./ia-v1.md)

## 1. 목적

이 문서는 v1 출시를 위해 필요한 작업을 우선순위와 의존성 기준으로 정리한 실행 백로그다.
각 항목은 GitHub 이슈로 바로 분리할 수 있는 수준을 목표로 한다.

## 2. 릴리스 전략

v1 백로그는 아래 4개 묶음으로 진행한다.

### Track A. 제품 기준선 정리

- 제품명과 용어 통일
- 상태 모델과 우선순위 규칙 정의

### Track B. 핵심 운영 대응 UX

- 상단 운영 카드 개편
- `Needs Attention` 추가
- 세션 작업 공간을 메인 진단 영역으로 승격
- alert에서 세션 상세로 바로 연결

### Track C. 신뢰성과 온보딩

- 연결 상태와 오류 상황 표시 강화
- 빈 상태와 첫 실행 안내
- stuck, cost spike 같은 운영 탐지 규칙 보강

### Track D. 폴리시와 성능 마감

- 분석 영역 재배치
- 세션 정렬과 필터 고도화
- 대량 이벤트 렌더링 성능 및 검증

## 3. 우선순위 기준

- `P0`: v1 출시 차단 항목
- `P1`: v1 완성도에 강하게 영향 주는 항목
- `P2`: v1 이후로 넘겨도 되는 확장 항목

## 5. 실행 순서

권장 순서는 아래와 같다.

1. `V1-01` 제품명과 용어 통일
2. `V1-02` 세션 상태 모델과 위험도 규칙 정의
3. `V1-03` 상단 요약 카드 개편
4. `V1-04` Needs Attention 추가
5. `V1-05` Sessions Workspace 재구성
6. `V1-06` alert to session 연결
7. `V1-07` 연결 상태 UX 보강
8. `V1-08` 빈 상태와 온보딩
9. `V1-09` stuck / cost spike 탐지
10. `V1-10` 분석 영역 재배치
11. `V1-11` 세션 정렬과 필터 개선
12. `V1-12` 성능 및 검증 마감

`V1-12`는 마지막에만 하는 작업이 아니라, 중간부터 계속 병행해도 된다.

## 6. 상세 백로그

### V1-01 통일하다: 제품명과 주요 UI 라벨

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `documentation`

배경:
현재 저장소와 실행 문구는 `Claude Code Monitor`인데, UI 제목은 `Claude Pulse`다.
제품 정체성이 흔들리면 이후 문서, 배포, 소개 문구, 스크린샷까지 모두 흔들린다.

산출물:

- 단일 제품명 확정
- 헤더, HTML title, README, 데스크톱 실행명, 문서 표현 통일
- 보조 카피 1줄 정의

완료조건:

- UI와 문서에서 제품명이 하나로 통일된다.
- 제품 설명 문장이 README와 첫 화면에서 같은 방향을 유지한다.
- 더 이상 `Monitor`와 `Pulse`가 혼용되지 않는다.

선행 의존성:

- 없음

### V1-02 정의하다: 세션 상태 모델과 위험도 규칙

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `enhancement`

배경:
현재 `ok`, `warning`, `error`만으로는 운영 대응 우선순위를 만들기 어렵다.
`Needs Attention`, 세션 정렬, 카드 지표는 모두 공통 상태 모델이 있어야 동작한다.

제안 초안:

- `active`
- `idle`
- `stuck`
- `completed`
- `failed`

초기 working rule:

- `failed`: 최근 이벤트 또는 집계 상태에 `error` 포함
- `stuck`: 최근 일정 시간 동안 이벤트 없음, 아직 완료로 보기 어려움
- `active`: 최근 활동이 임계값 이내
- `idle`: 최근 활동은 없지만 실패는 아님
- `completed`: 장시간 활동 없음 + 종료된 흐름으로 판단 가능

산출물:

- 상태 모델 정의 문서
- 위험도 정렬 규칙
- 프론트엔드와 백엔드가 공통으로 쓸 판정 기준

완료조건:

- 임의의 세션을 보면 어떤 상태로 분류되는지 설명 가능하다.
- 세션 목록, alerts, cards, workflow가 같은 상태 모델을 쓴다.
- 위험도 우선순위가 명시된다.

선행 의존성:

- 없음

### V1-03 개편하다: 상단 요약 카드를 세션 중심 운영 지표로 재구성

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `enhancement`

배경:
현재 카드는 `Active`, `Error`, `Sessions`, `Tokens`, `Cost` 중심이다.
v1에서는 사용자가 "지금 어디가 문제인가"를 먼저 봐야 한다.

권장 카드 구성:

- Active Sessions
- Needs Attention
- Total Sessions
- Range Tokens

산출물:

- 카드 구성 재설계
- 카드 라벨 문구 정리
- 세션 중심 지표 기준 반영

완료조건:

- 첫 화면 카드만 보고 활성 규모와 위험 규모를 파악할 수 있다.
- `Needs Attention` 수치가 일관된 규칙으로 계산된다.
- 카드의 단위가 agent 중심에서 session 중심으로 이동한다.

선행 의존성:

- `V1-02`

### V1-04 추가하다: Needs Attention 섹션과 우선순위 정렬

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `enhancement`

배경:
v1 IA의 핵심은 "문제 세션을 페이지 상단에서 바로 보여주는 것"이다.
이 영역이 없으면 사용자는 차트와 로그 사이를 직접 탐색해야 한다.

표시 규칙:

- failed 세션 우선
- stuck 세션 다음
- warning 세션 다음
- cost spike 세션 다음

각 행 필수 정보:

- 세션 ID 또는 표시 이름
- 상태
- 마지막 이벤트
- 마지막 활동 시각
- 토큰
- 비용
- 참여 agent 수

완료조건:

- 첫 화면 상단에서 문제 세션을 바로 볼 수 있다.
- 정렬 규칙이 문서화되고 재현 가능하다.
- 클릭 시 세션 상세로 진입한다.

선행 의존성:

- `V1-02`
- `V1-03`

### V1-05 재구성하다: Sessions Workspace를 메인 진단 영역으로 배치

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `enhancement`

배경:
현재 세션 리스트와 상세는 존재하지만 메인 작업면으로 보이지 않는다.
v1에서는 세션 목록과 상세가 진단의 중심이어야 한다.

산출물:

- 세션 목록을 상단 진단 구역으로 이동
- 선택 세션 상세를 메인 패널로 유지
- 세션 헤더 정보 보강
  - 상태, 마지막 활동, 토큰, 비용, agent 수

완료조건:

- 사용자는 세션 목록에서 바로 상세로 들어간다.
- 선택된 세션은 스냅샷 갱신 중에도 유지된다.
- 세션 상세가 숨겨진 보조 UI가 아니라 메인 진단 UI로 작동한다.

선행 의존성:

- `V1-02`
- `V1-04`

### V1-06 연결하다: alert에서 session detail로 바로 진입

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `enhancement`

배경:
현재 alert drilldown은 agent 기준 맥락이 강하다.
v1의 대표 진입점은 agent보다 session이어야 한다.

산출물:

- alert 클릭 시 관련 session detail 열기
- 동일 세션 반복 경고를 묶는 규칙 초안
- alert와 session 간 연결 표시

완료조건:

- 경고 확인 후 2클릭 이내에 관련 세션 상세로 이동한다.
- alert에서 해당 세션의 최근 이벤트를 바로 볼 수 있다.
- agent 기준 drilldown이 있더라도 session 진입이 우선 동작한다.

선행 의존성:

- `V1-05`

### V1-07 보강하다: 연결 끊김과 재연결 상태 UX

- 우선순위: `P0`
- 출시 차단: `Yes`
- 제안 라벨: `status: ready`, `priority: high`, `bug`

배경:
모니터링 도구에서 데이터가 비어 있는 이유가 "정상 무이벤트"인지 "수집 실패"인지 구분되지 않으면 신뢰가 무너진다.

산출물:

- 연결 상태 UX 강화
- 마지막 성공 시각 노출
- 오프라인 또는 재연결 상태에서의 안내 문구
- 수집 실패와 UI 연결 실패의 구분

완료조건:

- 사용자가 데이터 공백의 원인을 추측하지 않아도 된다.
- offline, reconnecting, connected 상태가 명확히 보인다.
- 연결 실패 시 다음 행동이 안내된다.

선행 의존성:

- 없음

### V1-08 추가하다: 첫 실행 빈 상태와 수집 경로 안내

- 우선순위: `P1`
- 출시 차단: `No`
- 제안 라벨: `status: ready`, `priority: medium`, `enhancement`

배경:
데이터가 없는 첫 실행 상태는 지금의 UI에서 가장 불친절한 순간일 가능성이 높다.

- 빈 상태 전용 메시지
- 데이터 위치 안내
- 수집 주기 또는 새로고침 힌트
- 필요 시 샘플 데이터 또는 예시 화면 방향성

완료조건:

- 신규 사용자가 "왜 비어 있는지"와 "다음에 무엇을 해야 하는지"를 이해한다.
- 빈 상태가 오류 화면처럼 보이지 않는다.

선행 의존성:

- `V1-07`

### V1-09 탐지하다: stuck 세션과 cost spike 규칙

- 우선순위: `P1`
- 출시 차단: `No`
- 제안 라벨: `status: ready`, `priority: medium`, `enhancement`

배경:
warning/error만으로는 사용자가 실제로 개입해야 할 상황을 다 잡아내기 어렵다.

초기 working rule:

- stuck: 최근 2분 이상 이벤트 없음, completed가 아닌 세션
- cost spike: 짧은 구간 비용 또는 토큰 증가율이 임계치를 넘는 세션

산출물:

- stuck 규칙
- cost spike 규칙
- Needs Attention 및 카드에 반영할 계산 로직

완료조건:

- warning/error 외에도 운영상 위험 세션이 식별된다.
- 탐지 기준이 문서화된다.
- false positive를 줄이기 위한 초기 안전장치가 있다.

선행 의존성:

- `V1-02`
- `V1-03`
- `V1-04`

### V1-10 정리하다: workflow, timeline, graphs를 보조 분석 영역으로 재배치

- 우선순위: `P1`
- 출시 차단: `No`
- 제안 라벨: `status: ready`, `priority: medium`, `enhancement`

배경:
현재는 분석 패널이 진단 패널보다 위에 있다.

산출물:

- 모바일 우선순위 재조정
- 분석 패널과 진단 패널의 역할 문구 정리

완료조건:

- 문제 세션을 찾기 위해 차트부터 볼 필요가 없다.
- 그래프와 타임라인은 보조 분석 영역으로 위치가 정리된다.

선행 의존성:

- `V1-04`
- `V1-05`

### V1-11 개선하다: 세션 정렬과 필터를 위험도 중심으로 고도화

- 우선순위: `P1`
- 출시 차단: `No`
- 제안 라벨: `status: ready`, `priority: medium`, `enhancement`

배경:
세션 수가 많아질수록 정렬 규칙과 필터가 중요해진다.
문제 세션이 묻히면 v1 가치가 약해진다.

산출물:

- 기본 정렬 규칙 고도화
- 위험도, 최근 활동, 비용 기준 정렬
