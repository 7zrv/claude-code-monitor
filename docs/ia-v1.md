# Claude Code Monitor v1 정보구조와 화면 우선순위

- 문서 상태: Draft
- 작성일: 2026-03-08
- 기준 문서: [prd-v1.md](./prd-v1.md)

## 1. 문서 목적

v1에서 사용자가 첫 화면에서 무엇을 먼저 보고, 어떤 흐름으로 상세 진단에 들어가야 하는지 정의한다.
이 문서는 기능 목록이 아니라 화면 우선순위와 탐색 구조를 결정하는 문서다.

## 2. 핵심 원칙

- 첫 화면은 분석보다 운영 대응을 우선한다.
- 기본 단위는 `event`보다 `session`을 우선한다.
- 사용자는 5초 안에 "지금 개입이 필요한가"를 판단할 수 있어야 한다.
- 상세 정보는 많아도 되지만, 1차 화면은 우선순위가 명확해야 한다.
- 차트는 보조 수단이고, 문제 세션 식별이 주 기능이다.

## 3. 첫 5초에 답해야 하는 질문

사용자가 대시보드를 열었을 때 바로 답해야 하는 질문은 아래 네 가지다.

1. 지금 수집이 정상 연결 상태인가
2. 현재 활성 세션은 몇 개인가
3. 지금 개입이 필요한 세션은 무엇인가
4. 비용 또는 오류가 비정상적으로 증가하고 있는가

이 질문에 바로 답하지 못하면, 이후의 차트와 로그는 제품 가치를 보강하지 못한다.

## 4. 현재 화면 구조 진단

현재 화면은 대체로 아래 순서로 구성돼 있다.

1. 헤더
2. 시간 범위 버튼
3. 요약 카드
4. 세션 타임라인
5. 그래프 3종
6. 워크플로우 진행 현황
7. 세션 드릴다운
8. 에이전트 테이블
9. 최근 이벤트
10. alerts

이 구조의 문제는 다음과 같다.

- 차트와 타임라인이 너무 앞에 있어 "무엇이 문제인지"보다 "무슨 데이터가 있는지"가 먼저 보인다.
- 세션 드릴다운이 중간 이후에 있어 진단 진입 경로가 약하다.
- alerts가 하단에 있어 운영 대응 우선순위가 낮게 느껴진다.
- agent, workflow, event, alert가 병렬로 나열돼 있어 대표 단위가 불명확하다.

## 5. v1 정보구조

v1은 하나의 메인 화면 안에서 아래 5개 레이어로 정보를 구성한다.

### Layer 1. 상태 확인

사용자가 대시보드를 열자마자 시스템이 살아 있는지 확인하는 영역.

- 제품명
- 연결 상태
- 마지막 갱신 시각
- 시간 범위 선택
- 핵심 요약 카드

### Layer 2. 개입 우선순위

지금 당장 봐야 하는 세션을 우선 노출하는 영역.

- `Needs Attention` 세션 목록
- 실패, 경고, stuck, 비용 급증 세션 우선 정렬
- 각 행에서 세션 상세로 바로 진입

### Layer 3. 세션 작업 공간

운영 대응의 중심이 되는 영역.

- 세션 목록
- 선택된 세션의 상세 이벤트 흐름
- 세션 단위 토큰, 비용, 참여 agent 수, 마지막 활동 시각

### Layer 4. 보조 진단

상세 원인 파악에 도움이 되는 보조 영역.

- alerts
- workflow 진행 현황
- agent 상태 테이블

### Layer 5. 분석 및 탐색

상황을 넓게 해석하거나 회고할 때 쓰는 영역.

- 세션 타임라인
- 처리량, 토큰 추이, 도구 호출 그래프
- 최근 이벤트 탐색

## 6. 권장 화면 순서

### 6.1 Desktop

v1 메인 화면의 권장 순서는 아래와 같다.

1. Header
2. Time range + connection state + last refresh
3. Summary cards
4. Sessions workspace
5. Needs Attention
6. Alerts
7. Workflow
8. Agents
9. Timeline
10. Graphs
11. Recent events

### 6.2 Mobile

모바일에서는 세로 스택을 더 강하게 가져간다.

1. Header
2. Connection / refresh / time range
3. Summary cards
4. Sessions list
5. Session detail
6. Needs Attention
7. Alerts
8. Workflow
9. Agents
10. Timeline
11. Graphs
12. Recent events

모바일에서는 차트보다 세션 리스트와 상세를 우선 고정해야 한다.

## 7. 화면별 정의

### 7.1 Header

목적:
현재 제품 상태와 연결 상태를 즉시 전달한다.

필수 요소:

- 제품명
- 제품 설명 1줄
- 연결 상태 배지
- 마지막 갱신 시각

설계 메모:

- 연결 상태는 장식이 아니라 운영 신뢰의 핵심이다.
- 상태 텍스트는 항상 보여야 한다.

### 7.2 Summary Cards

목적:
현재 상태를 숫자로 압축해 보여준다.

v1 필수 카드:

- Active Sessions 또는 Active Agents
- Needs Attention Count
- Total Sessions
- Range Tokens
- Range Cost

권장 변경:

- 현재 `Error` 카드만으로는 부족하다.
- `Error` 대신 또는 함께 `Needs Attention` 카드가 필요하다.
- 핵심 단위를 `agent`보다 `session` 중심으로 바꾸는 것이 바람직하다.

### 7.3 Sessions Workspace

목적:
운영자가 가장 오래 머무는 핵심 작업 공간.

구성:

- 좌측: 세션 목록
- 우측: 선택 세션 상세

세션 목록 기본 정렬:

1. 문제 상태 세션
2. 최근 활동 세션
3. 토큰 또는 비용이 큰 세션

세션 목록 기본 표시:

- 세션 ID
- 상태
- 마지막 활동
- 토큰
- 비용
- agent 수

세션 상세 표시:

- 세션 헤더
- 세션 상태
- 최근 이벤트 흐름
- 관련 alerts
- 세션 참여 agents

설계 메모:

- v1의 대표 탐색 흐름은 `세션 선택 -> 이벤트 확인 -> 원인 파악`이다.
- 따라서 세션 상세는 별도 숨겨진 보조 화면이 아니라 메인 작업면이어야 한다.

### 7.4 Needs Attention

목적:
지금 개입해야 할 대상을 세션 작업면 바로 아래에서 빠르게 알려준다.

표시 대상:

- `error` 포함 세션
- 최근 일정 시간 동안 응답이 없는 `stuck` 후보 세션
- `warning` 누적 세션
- 비용이 짧은 시간에 급증한 세션

행 정보:

- 세션 ID 또는 표시 이름
- 현재 상태
- 마지막 이벤트
- 마지막 활동 시각
- 토큰 / 비용
- 참여 agent 수

행 액션:

- 클릭 시 세션 상세 열기
- 경고 사유 또는 대표 이벤트 노출

정렬 규칙:

1. failed
2. stuck
3. warning
4. cost spike
5. recently active

### 7.5 Alerts

목적:
문제 이벤트를 짧고 명확하게 보여준다.

v1 역할:

- 알림 자체보다 진단 진입점 역할이 더 중요하다.
- alert 클릭 시 agent drilldown뿐 아니라 session detail 연결이 가능해야 한다.

표시 우선순위:

1. error
2. warning
3. 동일 세션 반복 경고는 묶어서 표시

현재 대비 보강 포인트:

- 현재는 agent 기준 drilldown이 강하다.
- v1에서는 session 기준 진입이 기본이어야 한다.

### 7.6 Workflow

목적:
실행 흐름과 완료/진행 상태를 빠르게 훑게 해준다.

위치 원칙:

- 세션 작업 공간 아래, 차트 위
- 1차 대응 화면이 아니라 보조 상황판 역할

표시 원칙:

- 활성 세션과 완료 세션을 명확히 분리
- 완료 세션은 기본 축소 가능

### 7.7 Agents

목적:
세션이 아니라 개별 agent 상태를 확인할 때 사용한다.

위치 원칙:

- 메인 진단 흐름보다 아래
- power user용 보조 표면

설계 메모:

- agent 표는 중요하지만, 홈 화면의 대표 단위가 되어서는 안 된다.
- 계층형 트리는 v1에 들어가더라도 세션 중심 구조를 깨지 않는 범위에서 써야 한다.

### 7.8 Timeline

목적:
세션 또는 agent 활동 패턴을 시각적으로 파악한다.

위치 원칙:

- 요약과 진단 뒤에 배치한다.
- 차트와 함께 분석 영역에 둔다.

설계 메모:

- 타임라인은 "보면 유용한 것"이지 "먼저 봐야 하는 것"은 아니다.

### 7.9 Graphs

목적:
이벤트량, 토큰 사용, 도구 호출 패턴을 분석한다.

위치 원칙:

- 메인 대응 화면 뒤
- 회고와 분석 성격

v1 판단:

- 유지 가치가 충분하지만 우선순위는 낮다.
- 상단 배치 대상이 아니다.

### 7.10 Recent Events

목적:
원시 로그 탐색과 검색을 지원한다.

위치 원칙:

- 페이지 하단
- 고급 탐색 영역

설계 메모:

- 최근 이벤트는 강력한 기능이지만, v1에서 홈 화면의 주인공이 되어서는 안 된다.

## 8. 대표 사용자 흐름

### 8.1 운영 점검 흐름

1. 사용자가 화면을 연다
2. 연결 상태와 요약 카드를 본다
3. `Sessions Workspace`에서 최근 세션과 선택 가능한 작업면을 확인한다
4. `Needs Attention`에서 우선 개입 대상을 빠르게 좁힌다
5. 필요한 세션 상세로 진입한다

### 8.2 문제 진단 흐름

1. 사용자가 alert 또는 Needs Attention 항목을 클릭한다
2. 해당 session detail이 열린다
3. 최근 이벤트와 마지막 상태를 본다
4. 필요 시 agent 표와 raw events에서 추가 확인한다

### 8.3 비용 점검 흐름

1. 시간 범위를 바꾼다
2. range token / cost를 본다
3. 비용 급증 세션을 찾는다
4. 세션 상세에서 원인 이벤트를 확인한다

## 9. 기본 상호작용 규칙

- 선택된 세션은 새 스냅샷이 와도 유지된다.
- alert 클릭은 가능한 한 세션 상세를 연다.
- 필터 상태는 로컬에 저장한다.
- 완료 세션은 기본 접기 허용, 문제 세션은 기본 펼침
- 검색은 raw events뿐 아니라 세션 목록에도 확장하는 것이 바람직하다.

## 10. 기본 정렬 규칙

### 10.1 Sessions

기본 정렬 우선순위:

1. failed
2. stuck
3. warning
4. recently active
5. high cost
6. latest updated

### 10.2 Alerts

기본 정렬 우선순위:

1. severity
2. recency
3. repeated occurrence

### 10.3 Agents

기본 정렬 우선순위:

1. session risk
2. recency
3. token usage

## 11. 빈 상태 설계

v1에서 빈 상태는 반드시 별도 설계가 필요하다.

첫 실행 시 보여줄 내용:

- 아직 수집된 세션이 없다는 메시지
- 수집 경로 설명
- 예상 데이터 위치
- 수집 주기 또는 새로고침 힌트

연결 실패 시 보여줄 내용:

- 현재 상태
- 마지막 성공 시각
- 가능한 원인 1~2개
- 다음 확인 행동

## 12. 현재 구조에서 바로 바꿔야 할 것

### Must

- alerts를 페이지 하단에서 진단 영역으로 올린다.
- sessions 영역을 그래프보다 위로 올린다.
- `Needs Attention` 영역을 새로 만든다.
- 요약 카드에 `Needs Attention` 또는 동등한 운영 지표를 추가한다.

### Should

- 최근 이벤트 영역을 페이지 하단 탐색 영역으로 고정한다.
- workflow를 세션 중심 진단을 보조하는 위치로 내린다.
- 타임라인과 그래프를 분석 영역으로 묶는다.

### Could

- 세션 상세를 우측 고정 패널 형태로 바꾼다.
- agent 표를 세션 상세 안의 보조 탭으로 흡수한다.

## 13. v1 완료 기준

아래가 충족되면 화면 정보구조 측면에서 v1 기준을 만족한 것으로 본다.

- 첫 화면에서 연결 상태, 활성 상태, 문제 세션, 비용 상태가 순서대로 보인다.
- 사용자는 문제 세션을 찾기 위해 페이지 하단까지 스크롤할 필요가 없다.
- 세션 목록과 세션 상세가 메인 진단 흐름을 담당한다.
- alerts는 세션 진단의 입구 역할을 한다.
- 차트와 raw events는 보조 분석 역할로 위치가 정리된다.
