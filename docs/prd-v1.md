# Claude Code Monitor v1 PRD

- 문서 상태: Draft
- 작성일: 2026-03-08
- 작성 목적: 제품 방향, 출시 기준, 우선순위를 명확히 정의한다.

## 1. 제품 한 줄 정의

Claude Code 세션과 에이전트를 여러 개 동시에 실행하는 개발자를 위해,
실시간 상태, 실패 징후, 토큰, 비용을 한눈에 파악하게 해주는 로컬 퍼스트 운영 대시보드.

## 2. 배경

현재 프로젝트는 로컬 Claude 데이터 디렉토리를 자동 수집하고, Rust 서버와 Electron UI를 통해 다음 정보를 보여준다.

- 실시간 이벤트 스냅샷과 SSE 스트림
- 에이전트별 상태와 최근 이벤트
- 세션 드릴다운
- 경고 및 오류 알림
- 토큰 및 비용 지표
- 워크플로우 진행 현황

기능은 이미 존재하지만, 제품 관점에서는 아직 "무엇을 가장 먼저 해결하는 도구인가"가 완전히 고정되지 않았다.
이 문서는 v1의 제품 정체성을 "개발자용 로컬 운영 콘솔"로 정리한다.

## 3. 문제 정의

Claude Code를 병렬로 실행하는 사용자는 아래 문제를 자주 겪는다.

- 지금 어떤 세션이 실행 중인지 즉시 알기 어렵다.
- 실패했거나 멈춘 세션을 늦게 발견한다.
- 특정 오류가 어느 세션과 어느 이벤트에서 발생했는지 찾는 데 시간이 걸린다.
- 토큰과 비용이 세션 단위로 어떻게 쌓이는지 직관적으로 보이지 않는다.
- 작업이 끝난 후 회고할 수 있는 운영 기록이 분산되어 있다.

## 4. 목표 사용자

### 4.1 핵심 사용자

- 하나의 머신에서 여러 Claude Code 세션을 동시에 돌리는 개발자
- 장시간 작업, 자동화 작업, 서브에이전트 작업을 관리해야 하는 AI 엔지니어
- 비용과 실패를 직접 통제해야 하는 개인 개발자 또는 소규모 팀

### 4.2 제외 사용자

- 사내 다수 인원이 함께 쓰는 중앙 집중형 관제 시스템을 원하는 조직
- 권한 관리, 계정 관리, 클라우드 동기화가 필수인 팀
- Claude 외 여러 AI 도구를 단일 플랫폼에서 통합 관제하려는 사용자

## 5. Jobs To Be Done

- 내가 지금 돌리고 있는 Claude 작업 중 어디에 개입해야 하는지 빠르게 알고 싶다.
- 실패나 경고가 생기면 원인이 되는 세션과 이벤트를 바로 열고 싶다.
- 오늘 어느 세션이 토큰과 비용을 많이 썼는지 확인하고 싶다.
- 세션이 끝난 뒤에도 어떤 흐름으로 진행됐는지 회고하고 싶다.

## 6. 제품 목표

### 6.1 비즈니스 목표

- Claude Code 사용자의 운영 피로를 줄이는 데 집중한 명확한 유틸리티 제품으로 자리 잡는다.
- "모니터링이 필요한 Claude 파워유저"에게 기본 도구로 인식된다.

### 6.2 사용자 목표

- 5초 안에 활성, 지연, 실패 상태를 식별한다.
- 2클릭 이내에 alert에서 문제 세션 상세로 진입한다.
- 세션별 토큰과 비용 사용량을 같은 화면 흐름 안에서 확인한다.

### 6.3 성공 지표

- Time to detect: 문제 세션을 발견하는 시간 5초 이내
- Time to diagnose: 경고 확인 후 관련 세션 상세 진입 2클릭 이내
- Session visibility: 활성 세션의 90% 이상이 식별 가능한 상태로 표시됨
- Token visibility: 상위 비용 세션을 사용자가 별도 로그 탐색 없이 찾을 수 있음

## 7. 비목표

v1에서는 아래를 하지 않는다.

- 멀티 유저 계정 시스템
- 클라우드 동기화 및 원격 서버 수집
- 권한 관리와 역할 기반 접근 제어
- Slack, Discord, 이메일 등 외부 알림 연동
- 세션 제어 기능
  - 예: 중지, 재시도, 승인, 프롬프트 재실행
- Claude 외 타 에이전트 플랫폼 통합

## 8. 제품 원칙

- 로컬 우선: 사용자의 로컬 환경만으로 동작해야 한다.
- 즉시성 우선: 상세 분석보다 먼저 "지금 어디가 문제인가"를 드러내야 한다.
- 세션 중심: 이벤트 나열보다 세션 상태와 흐름 파악을 우선한다.
- 저설정: 별도 복잡한 설정 없이 실행 가능해야 한다.
- 회고 가능성: 실시간뿐 아니라 최근 히스토리도 추적할 수 있어야 한다.

## 9. 현재 기준선

현재 구현 기준으로 v1 출발선에 포함되는 기능은 다음과 같다.

- 로컬 Claude 데이터 자동 수집
- 이벤트 스냅샷 API
- SSE 실시간 스트림
- alerts 목록
- 세션별 이벤트 조회
- 대시보드 카드
- 타임라인, 이벤트 처리량, 토큰 추이, 도구 호출 통계
- 워크플로우 진행 현황
- 세션 리스트 및 상세
- agent 필터, event 필터, 검색
- SQLite 기반 시간 버킷 저장과 보존 기간 관리
- Electron 실행 방식

## 10. v1 핵심 사용 시나리오

### 10.1 운영 확인

사용자는 대시보드를 열자마자 아래를 알아야 한다.

- 지금 몇 개의 세션이 활성 상태인지
- 어떤 세션이 경고 또는 오류 상태인지
- 어떤 에이전트가 마지막으로 언제 이벤트를 보냈는지

### 10.2 문제 진단

사용자는 alert 또는 세션 목록에서 문제 세션을 선택하고, 해당 세션의 최근 이벤트 흐름을 확인할 수 있어야 한다.

### 10.3 비용 점검

사용자는 최근 1시간, 1일, 1주 범위에서 토큰과 비용 변화를 보고, 세션 또는 에이전트 단위로 원인을 좁힐 수 있어야 한다.

### 10.4 작업 회고

사용자는 완료된 세션과 워크플로우를 다시 보며, 어느 시점에 어떤 상태 변화가 있었는지 확인할 수 있어야 한다.

## 11. v1 필수 요구사항

### 11.1 데이터 수집

- 사용자의 Claude 로컬 데이터에서 이벤트를 자동 수집한다.
- 초기 로드 시 최근 히스토리를 일정 범위까지 백필한다.
- 수집 실패 시 UI에서 연결 상태를 명확히 보여준다.

### 11.2 상태 가시성

- 연결 상태를 `connected`, `reconnecting`, `offline`으로 표시한다.
- 에이전트별 최근 상태, 마지막 이벤트, 마지막 활동 시점을 제공한다.
- 세션별 목록과 세션 상세 이벤트 흐름을 제공한다.

### 11.3 경고 및 탐지

- 최소한 `warning`, `error` 이벤트는 alert로 surfaced 된다.
- alert에서 관련 세션으로 빠르게 진입할 수 있어야 한다.
- 사용자는 현재 개입이 필요한 대상을 우선적으로 인지할 수 있어야 한다.

### 11.4 사용량 가시성

- 총 토큰, 총 비용, 에이전트별 토큰을 제공한다.
- 시간 범위별 집계를 제공한다.
- 사용량 급증 여부를 시각적으로 파악할 수 있어야 한다.

### 11.5 탐색 및 필터링

- 상태별 필터, 행 수 제한, 키워드 검색을 제공한다.
- 사용자가 선택한 필터와 토글 상태는 재방문 시 복원된다.

### 11.6 보존 및 성능

- 최근 사용량 버킷은 로컬 DB에 저장한다.
- 기본 보존 기간은 90일로 둔다.
- 일반적인 로컬 사용 환경에서 대시보드 첫 로드는 지연 없이 체감 가능해야 한다.

## 12. v1 출시 전에 보강할 요구사항

다음 항목은 현재 구현과 별개로, 제품 완성도를 위해 v1 전에 우선 검토해야 한다.

### 12.1 상태 모델 정교화

- 현재 `ok`, `warning`, `error`만으로는 "멈춤"을 설명하기 어렵다.
- `active`, `idle`, `stuck`, `completed`, `failed` 같은 운영 상태 모델이 필요하다.

### 12.2 세션 중심 우선순위 재배치

- 화면 정보량은 많지만, "지금 가장 위험한 세션"이 첫 시선에서 바로 드러나는 구조는 아니다.
- 첫 화면 상단에 문제 세션 우선 영역을 두는 개선이 필요하다.

### 12.3 네이밍 일관성

- 저장소와 실행명은 `Claude Code Monitor`이고, UI 제목은 `Claude Pulse`다.
- v1 이전에 제품명을 하나로 통일해야 한다.

### 12.4 빈 상태와 온보딩

- 실제 데이터가 없을 때 무엇을 해야 하는지 안내가 필요하다.
- 첫 실행 가이드, 수집 경로 상태, 데모 데이터 여부를 보여줄 필요가 있다.

## 13. 우선순위

### P0

- 실시간 연결 안정성
- 세션 목록과 세션 상세 진입
- 경고와 오류 가시성
- 토큰 및 비용 카드
- 시간 범위 필터

### P1

- stuck 탐지 규칙
- 문제 세션 우선 노출 영역
- 온보딩 및 빈 상태 개선
- 네이밍 통일

### P2

- 계층형 agent 트리
- 사용자 정의 alert 규칙
- 데이터 export

## 14. 리스크

- Claude 로컬 데이터 포맷 변화 시 수집기가 깨질 수 있다.
- 이벤트 수가 많아질수록 UI 렌더링 성능 문제가 생길 수 있다.
- 로컬 전용 제품은 배포는 쉽지만 협업 기능 요구가 생기면 구조 변경 폭이 커진다.
- "이벤트를 많이 보여주는 것"과 "문제를 빨리 찾게 하는 것" 사이에서 제품 초점이 흐려질 수 있다.

## 15. 오픈 질문

- v1의 대표 단위는 `agent`인가 `session`인가
- `stuck`을 몇 분 무응답으로 정의할 것인가
- 비용 급증 alert 임계치는 절대값 기준인가 변화율 기준인가
- 완료 세션을 기본으로 숨길 것인가 보여줄 것인가
- 계층형 agent 트리를 v1 범위에 넣을 것인가 v1.1로 미룰 것인가

## 16. 출시 기준

아래 조건을 만족하면 v1 출시 후보로 본다.

- 로컬 수집이 기본 환경에서 안정적으로 동작한다.
- 사용자는 대시보드 첫 화면에서 활성 세션, 문제 세션, 총 토큰, 총 비용을 확인할 수 있다.
- alert에서 세션 상세까지 자연스럽게 이동할 수 있다.
- 최근 이벤트 검색과 상태 필터가 동작한다.
- 연결 끊김 상태가 명확히 표시된다.
- 기본 검증 명령과 테스트가 통과한다.
- 제품명과 주요 UI 라벨이 일관된다.
