# Claude Code Monitor v1 이슈 생성 순서표

- 작성일: 2026-03-08
- 기준 문서: `docs/issues-v1/README.md`, `docs/backlog-v1.md`
- 목적: v1 이슈를 GitHub에 어떤 순서로 등록할지 결정한다.

## 1. 원칙

- 먼저 등록할수록 의존성이 낮고, 이후 작업 기준이 되는 이슈여야 한다.
- 제품 기준선과 상태 모델이 정리되기 전에는 화면 개편 이슈를 과도하게 쪼개지 않는다.
- 같은 배치 안에서는 병렬 가능한 이슈만 함께 등록한다.
- `P2` 확장 이슈는 v1 출시 컷라인 이슈가 등록된 뒤 별도 트랙으로 올린다.

## 3. 배치별 생성 가이드

### Batch 1. 제품 기준선과 신뢰성

생성 순서:

1. `V1-01`
2. `V1-02`
3. `V1-07`

생성 이유:

- `V1-01`은 제품 이름과 용어를 고정한다.
- `V1-02`는 이후 거의 모든 화면/정렬/지표 이슈의 기준이 된다.
- `V1-07`은 독립성이 높아서 같은 시점에 등록해도 된다.

이 배치에서 멈춰야 하는 기준:

- 제품명이 확정되지 않았는데 UI 개편 이슈를 여럿 열기 시작하는 경우
- 상태 모델 없이 `Needs Attention`이나 정렬 이슈를 먼저 구현하려는 경우

### Batch 2. 핵심 진단 경험

생성 순서:

1. `V1-03`
2. `V1-04`
3. `V1-05`
4. `V1-06`

생성 이유:

- `V1-03` 카드 개편은 상태 모델을 사용자에게 노출하는 첫 단계다.
- `V1-04`는 v1 홈 화면의 핵심 차별점이므로 별도 이슈로 명확히 둔다.
- `V1-05`는 세션 중심 UX의 본체다.
- `V1-06`은 `V1-05`가 정리된 뒤 붙여야 흐름이 자연스럽다.

이 배치에서 멈춰야 하는 기준:

- 세션 작업 공간 구조가 없는 상태에서 alert 연결만 먼저 만들려는 경우
- 카드와 `Needs Attention`의 기준이 서로 다른 상태 모델을 쓰는 경우

### Batch 3. 완성도 보강

생성 순서:

1. `V1-08`
2. `V1-09`
3. `V1-10`
4. `V1-11`
5. `V1-12`

생성 이유:

- `V1-08`은 신뢰성과 온보딩을 보강한다.
- `V1-09`는 운영 탐지 정교화다.
- `V1-10`, `V1-11`은 진단 UX를 다듬는다.
- `V1-12`는 출시 차단이므로 늦지 않게 등록해 추적해야 한다.

이 배치에서 멈춰야 하는 기준:

- 성능 이슈를 등록하지 않은 채 UI 구조 변경만 계속 쌓이는 경우
- 온보딩과 연결 실패 경험이 방치된 채 차트 개선부터 시작하는 경우

### Batch 4. 후속 확장

생성 순서:

1. `V1-13`
2. `V1-14`
3. `V1-15`

생성 이유:

- 모두 유효한 기능이지만 v1 컷라인과 직접 연결되지는 않는다.
- v1가 안정되기 전에는 별도 트랙으로 관리하는 편이 낫다.

## 5. 병렬 진행 규칙

같이 진행 가능한 조합:

- `V1-01` + `V1-07`
- `V1-08` + `V1-10`
- `V1-10` + `V1-12`

부분 병렬 가능한 조합:

- `V1-05`와 `V1-11`
  - 레이아웃과 정렬 규칙 정의는 병렬 가능
  - 최종 구현은 `V1-05` 구조 확정 후 붙는 것이 안전
- `V1-04`와 `V1-09`
  - 탐지 규칙 설계는 병렬 가능
  - 최종 UI 노출은 `V1-04` 구조가 먼저 필요

같이 진행하면 안 되는 조합:

- `V1-02` 없이 `V1-03`, `V1-04`, `V1-11`
- `V1-05` 없이 `V1-06`
- `V1-09` 없이 `V1-14`
